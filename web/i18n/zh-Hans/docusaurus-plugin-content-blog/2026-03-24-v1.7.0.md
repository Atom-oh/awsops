## 多账户架构（PR #6）

通过 Steampipe Aggregator 模式，在单一仪表板中管理多个 AWS 账户。只需修改 `data/config.json` 中的 `accounts[]` 数组，即可在不改动代码的情况下添加账户。

<MultiAccountFlow />

| 组件 | 文件 | 作用 |
|----------|------|------|
| `AccountContext` | `src/components/` | 账户状态管理 + `useAccount()` Hook |
| `AccountSelector` | `src/components/` | 账户切换下拉菜单（支持 ARIA、键盘操作） |
| `AccountBadge` | `src/components/` | 按账户显示彩色圆点 + alias |
| `buildSearchPath()` | `src/lib/steampipe.ts` | 返回 `public, aws_{id}, kubernetes, trivy` |
| `runCostQueriesPerAccount()` | `src/lib/steampipe.ts` | 按账户执行 Cost 查询后以 `account_id` 标记并合并 |
| `/accounts` 页面 | `src/app/accounts/` | 账户添加/删除/测试 UI |
| `cross_account.py` | `agent/` | STS AssumeRole（credential 缓存、ExternalId、审计日志） |
| 配置脚本 | `scripts/11-setup-multi-account.sh` | Aggregator + 跨账户 IAM 角色自动化 |

:::info 代码变更范围
为全部 **25 个 SQL 查询**添加了 `account_id` 列，并在 **35 个页面**中应用了 `useAccountContext()`。DataTable 在检测到 `isMultiAccount && data[0].account_id` 时会自动添加 Account 列。
:::

## 安全修复（25 项）

按三级严重程度分类，共修复了 25 项安全问题。

:::warning CRITICAL 修复事项
在跨账户 AssumeRole 中强制启用**审计日志**和 **ExternalId**，防止权限提升和 Confused Deputy 攻击。
:::

| 严重程度 | 数量 | 主要项目 |
|--------|------|-----------|
| **CRITICAL** | 2 | AssumeRole 审计日志、强制启用 ExternalId |
| **HIGH** | 5 | Rate Limiting（5 req/min）、加强输入校验、`execFileSync` 安全 |
| **MEDIUM** | 18 | 防止 Pool exhaustion、连接销毁、错误消息过滤 |

**CRITICAL 详情**：

| 问题 | 风险 | 修复 |
|------|------|------|
| 缺少 AssumeRole 审计 | 无法追踪跨账户访问 | 在 `cross_account.py` 中记录所有 AssumeRole 调用 |
| 未使用 ExternalId | 可能遭受 Confused Deputy 攻击 | 为 AssumeRole 添加 ExternalId 必填参数 |

**HIGH 详情**：

| 问题 | 风险 | 修复 |
|------|------|------|
| 缺少 Rate Limiting | API 滥用、成本暴增 | 为 AI 路由应用 5 req/min 限制 |
| 输入校验不足 | 可能发生 SQL Injection | 对查询参数进行白名单校验 |
| 使用 `execFileSync` | Command Injection | 改为参数数组方式 |

## 实时 Bedrock 流式传输

以分块（chunk）方式实时传输 Bedrock 响应。

<BedrockStreamSequence />

| 项目 | Before (v1.6.0) | After (v1.7.0) |
|------|-----------------|-----------------|
| Bedrock 响应 | 一次性返回全部文本 | **按 chunk 实时传输** |
| API | `InvokeModelCommand` | `InvokeModelWithResponseStreamCommand` |
| 传输方式 | 完成后发送 `done` 事件 | 连续发送 `chunk` 事件 |
| 客户端 | 渲染完整文本 | 通过 `streamingContent` 状态实时渲染 |

:::info 流式传输辅助函数
`streamBedrockToSSE()` 函数（`src/app/api/ai/route.ts`）从 Bedrock 流中读取 chunk，并转换为 SSE `chunk` 事件。
:::

## 缓存预热

为提升仪表板加载速度，引入了后台缓存预热机制。

<CacheWarmerFlow />

| 项目 | 说明 |
|------|------|
| 查询数量 | 仪表板 23 个 + 监控 10 个 = **33 个** |
| 刷新周期 | **4 分钟**（在缓存 TTL 5 分钟到期前） |
| 初始化 | `ensureCacheWarmerStarted()` — lazy-init 模式 |
| 效果 | 首次加载页面时命中缓存，实现**即时响应** |

## 客户 Logo 自定义

在 `data/config.json` 中配置客户 Logo 后，无需重启服务器，1 分钟内即可生效。

```json
{
  "customerLogo": "/logos/customer-logo.png",
  "customerName": "Customer Corp",
  "customerLogoBg": "#ffffff"
}
```

- 将 Logo 文件放置在 `public/logos/` 目录下
- 仅通过修改 config 即可切换品牌 — 无需改动代码
