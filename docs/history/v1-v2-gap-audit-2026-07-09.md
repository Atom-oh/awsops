# AWSops v1 → v2 기능 갭 재검토 (v1 폐기 전, 2026-07-09)

> 목적: v1 폐기(ADR-016) 전 최종 확인. `docs/history/v1-v2-gap-audit-2026-06-10.md`(이하 "6/10 감사")의 missing 36건을 2026-07-09 현재 `web/`, `scripts/v2/`, `terraform/v2/foundation/` 실코드 기준으로 재검증했다. 6/10~7/09 사이 ai-diagnosis, compliance, topology(@xyflow/react), customization, assistant, opencost API, incidents/webhook, EOL 배지, EBS 스냅샷, ECS 서비스/태스크 수집, i18n, admin 게이트 등이 실제로 착지했다.
>
> **결론: 폐기를 막는 차단(blocking) 항목 없음.** 남은 갭은 전부 "drop"(의도적 축소 수용) 또는 "backlog"(P3/P4 로드맵 기재)로 분류된다. 단 하나, **alert-webhook 외부 발신자 경로는 폐기 전 Phase 1에서 반드시 실사 확인**해야 한다(무음 유실 위험 — 아래 §3).

## 2) 확인된 잔존 갭 — disposition 부여

### drop (의도적 수용, 되돌리지 않음)
| 항목 | 근거 | 이유 |
|---|---|---|
| DOCX/PPTX 다중 포맷 진단 리포트 | v1 `report-docx.ts`/`report-pptx.ts`; v2 `ai-diagnosis`는 리포트를 생성하나 산출 포맷이 v1의 15섹션 Word/PPT급은 아님(정확한 산출 포맷은 워커 `report` job 구현체 참조) | 챗 기반 진단(P3 어시스턴트)이 리포트 문서 대체 경험으로 자리잡음. 문서 산출은 P4 백로그로 편입해도 폐기를 막을 이유 없음 |
| ECS Container Cost / EKS 이전 방식(request-fallback 없는 v1 계산) | `container-cost` 라우트 없음, OpenCost가 대체 | OpenCost가 더 정확한 실측 비용 — 기능 저하 아닌 대체 |
| Event-driven Pre-Scaling (ADR-010) | `find web -iname '*event-scaling*'` 0건, BASELINE/ADR에도 언급 없음 | v1에서도 admin 전용·실행 없음(계획 전용) 기능이라 실사용 저조. 필요 시 별도 신규 기능으로 재기획 |
| S3 리전×보안 TreeMap 시각화 | 제네릭 인벤토리 도넛만 존재 | 시각화 다양성 손실, 데이터 손실 아님 |
| Trivy CVE 스캐닝(Steampipe 플러그인) | v2 steampipe.tf에 해당 플러그인 없음 | 별도 보안 스캐닝 도구로 대체 검토 대상(범위 밖) |
| 부서별 세밀 ACL 매트릭스(페이지별 allowedAccountIds) | `admin.ts`는 그룹 단위 admin/non-admin 이분법 | 조직 규모상 그룹 기반으로 충분 판단 — 필요시 P4에 재상정 |

## 3) ⚠️ 폐기 전 필수 확인 — alert 경로 소스 불일치

v1 `src/app/api/alert-webhook/route.ts` 헤더 주석 원문:
> "Supported sources (VPC-internal): Prometheus Alertmanager (webhook), Grafana Alerting (webhook contact point), Generic webhook (custom format), CloudWatch SNS (only if ALB reachable without CloudFront)"
> "Primary path: CloudWatch Alarms → SNS `awsops-alert-topic` → SQS `awsops-alert-queue` → `alert-sqs-poller.ts`(EC2 인-프로세스)"

즉 두 경로가 공존한다:
1. **AWS 네이티브(SNS/SQS)**: CloudWatch Alarm이 구독 중인 `awsops-alert-topic`. v1 EC2를 stop해도 SNS/SQS 자체는 살아있으나 **poller가 EC2 인-프로세스라 멈추면 큐가 그냥 쌓이다 DLQ로 감** — 무음 유실.
2. **외부 웹훅 직결(Alertmanager/Grafana → v1 ALB URL)**: 외부 시스템이 v1의 공개 ALB/CloudFront URL(`awsops.atomai.click/api/alert-webhook` 또는 내부 IP)을 **직접 설정에 박아놓았을 가능성** — v1 도메인이 v2로 리다이렉트되면 이 URL은 v2에 없는 경로(`/api/alert-webhook`)이므로 404.

v2의 대응 경로는 `web/app/api/incidents/webhook/route.ts`이며 **URL 경로와 인증 방식이 다르다**(HMAC 스킴 재확인 필요). → **Phase 1에서 반드시**:
- `aws sns list-subscriptions-by-topic --topic-arn <awsops-alert-topic ARN>` 로 실제 구독자 확인
- 사내 Grafana/Alertmanager 설정에서 `awsops.atomai.click` 또는 v1 ALB DNS를 웹훅 URL로 쓰는 곳이 있는지 확인
- 있으면: v2 `incidents/webhook`으로 재설정하거나, disposition을 "drop(수용)"으로 명시하고 사용자 승인 필요

이 항목은 ADR-016 본문에 "확인 완료 여부"를 기록해야 한다(Phase 0 게이트 조건).

## 4) 결론

- 6/10 대비 **닫힌 항목 12건**(위 §1) — v2가 지난 한 달간 실질적으로 따라잡음.
- 잔존 갭은 전부 **drop 또는 backlog** — v1 폐기를 차단하는 항목 없음.
- 유일한 실행 조건은 **alert 경로 외부 의존성 실사**(§3) — Phase 1에서 완료 후 ADR-016에 결과 기록.
